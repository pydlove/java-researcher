为什么不建议用数据库唯一性约束做幂等控制？

Java开发中的幂等控制最佳实践

#Java开发 #幂等性 #数据库优化 #分布式系统

墨墨说技术 · 干货分享

1. 性能瓶颈问题

高频请求下，数据库唯一性校验（如唯一索引）会引发大量锁竞争

在分布式系统中可能成为性能瓶颈

相比内存级或Redis的幂等控制，数据库操作开销更大

数据库唯一性约束在高并发场景下会成为系统瓶颈，影响整体吞吐量。

2. 错误处理 & 灵活性

错误处理复杂

违反约束会直接抛出SQL异常（如DuplicateKeyException）

需额外代码捕获并转换为业务友好的提示

灵活性不足

唯一约束通常基于固定字段（如订单号）

实际业务可能需要组合条件（如用户+操作类型+参数哈希）

数据库层面难以支持动态规则，而代码层可通过唯一请求ID或业务键灵活实现。

3. 事务 & 分布式问题

事务代价高

若幂等校验与其他写操作在同一个事务中，失败会导致整体回滚

独立幂等服务可拆分事务边界，例如先记录请求ID再处理业务

分布式场景局限

分库分表时，唯一约束可能失效（如仅分片内唯一）

需依赖分布式锁或全局存储（如Redis）实现跨节点幂等

分布式系统中，数据库唯一约束往往无法满足全局幂等性需求。

推荐替代方案

预生成唯一请求ID

（如雪花ID，客户端携带该ID）

使用Redis原子操作

（如SETNX key request_id EX 3600）

状态机设计

（如订单状态变更时校验前置状态）

短时间窗口内的请求参数去重

（如参数MD5+时间戳缓存）

这些方案在Java生态中可通过Spring AOP、自定义注解等优雅实现，兼顾性能与可维护性。
